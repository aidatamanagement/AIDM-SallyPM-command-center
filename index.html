How Sally works

The daily workflow that keeps all projects in sync

Sally is AIDM's AI project manager. Every morning, Samina runs /sally sync and the system handles everything in 4 steps. Sally orchestrates 5 project agents that each own a client's files. Asana reconciliation is a separate skill (/update-asana) that runs after file sync and can also be used by any team member from their personal agent.

1
Sally files meeting notes
Automatic - no approval needed

Sally scans the _Meeting notes/ folder for new transcripts. She reads each one, classifies it by client (CCV, VQH, Real You, etc.), splits multi-client meetings, and files the content into the right project records.

/process-meeting-notes
2
Project agents update documents
Automatic - no approval needed

Sally spawns each project agent to process the newly filed notes. Each agent reads the notes and updates its own spreadsheets, trackers, business context docs, and knowledge bases.

@CCV
Tracker, context, PRD
@VQH
Master tracker, context
@RealYou
Context, architecture
@VQInc
Context, knowledge
@AIDM
Internal tracker
/update-ccv /update-vqh /update-realyou /update-vqinc /update-aidm
3
Samina reviews the changelog
Human review - one interaction point

Sally presents a combined changelog of everything that was filed and updated. Samina reviews it, flags anything wrong, and Sally fixes it. This is the only approval step in the entire sync.

/sally audit /reflect
4
Asana reconciliation
Requires approval

Sally runs /update-asana to compare project files against Asana. She applies the deliverable test (only real deliverables become tasks, not meetings, emails, or conversations) and focuses on a two-week rolling window. She proposes fewer, higher-quality tasks with rich descriptions that track what was said in each meeting. Samina approves the batch before anything changes.

Team members can use this too. If you need to update Asana from your personal agent, ask it to run /update-asana. It follows the same rules as Sally: deliverable test, two-week window, meeting context in descriptions. All Asana operations go through this one skill.
/update-asana /update-asana-status /monday-emails

What we want Sally to do

The vision for Sally as a fully autonomous team member

Where Sally is today

Sally runs on-demand when Samina opens Claude and types a command. She cannot run on her own, respond to messages, or monitor anything in real time. Every action requires a human to start it. The goal below is to make her a 24/7 autonomous team member.

1
Auto-process meeting notes as meetings end
Not built yet

When a Google Meet recording lands in the _Meeting notes/ folder, Sally should automatically pick it up, classify it by client, file it into the right project records, and update all related documents (trackers, business context, knowledge bases). No human trigger needed. Today this requires Samina to manually run /sally sync every morning.

2
Daily morning briefing to Samina and Keith
Not built yet

Every morning (before 8 AM), Sally sends an email or Google Chat message to Samina and Keith with a summary: what happened yesterday, which tasks were completed, what documents were updated, any requests that came in, what's due today, and any blockers. A full daily standup from an AI team member, ready before anyone opens their laptop.

3
Respond to Google Chat and emails addressed to her
Not built yet

Team members should be able to message @Sally in Google Chat or email sally@ and get answers: "What's the status of CCV?", "When is Keith's next trip?", "Can you draft an email to Matt about the assessment?". Sally has access to all project files, Asana, Gmail, and Calendar. She should be able to answer in real time, not just when Samina has Claude open.

4
Create and manage Asana tasks from meeting notes
Built - needs autonomous execution

/update-asana is now a standalone skill. Sally compares project files against Asana, applies a deliverable test (only real deliverables, not meetings or emails), and focuses on a two-week window. Task descriptions include a meeting context tracker with exact details from each discussion. Today it still requires Samina to approve. The goal: Sally runs this automatically after every sync, with team members able to trigger it from their own agents too.

5
Learn from corrections in real time
Not built yet

If someone tells Sally she used the wrong name, filed a note to the wrong client, or made any mistake, she should fix it immediately and remember the correction. Today she has a playbook system that records lessons between sessions, but it requires manual reflection. The goal is: tell her once, she never makes that mistake again, and she fixes the original error on the spot.

6
Run scheduled tasks and generate reports on a timer
Not built yet

Weekly reports should generate every Monday morning. The command center dashboard should refresh daily. Project health audits should run weekly. Overdue task follow-ups should go out automatically. All of this exists as slash commands today, but someone has to type them. Sally should run them on a schedule, like a cron job with project intelligence.

The end state

Sally becomes a real team member who works 24/7. She processes information as it arrives, keeps everyone informed, handles routine project management tasks autonomously, and learns from feedback. The team interacts with her through Google Chat, email, and Asana, not through a CLI. Samina's role shifts from operating Sally to reviewing her work and making strategic decisions.

Your personal agent can answer questions about any project you're assigned to. If it needs cross-client data, it reads the same files Sally maintains. No information gets siloed.
